# Aritiq — Phase 3 Progress

*From independent claims to a connected verdict: a provenance graph, a disclosure-language scan, and a dependency-aware score — every one of them still pure, model-free code.*

## The one-paragraph version

Phase 1 built the firewall: an LLM turns messy financial prose into structured claims, and deterministic code with no model in it verifies the arithmetic. Phase 2 widened the deterministic zone — a document registry, parsed tables, time series, six new but still model-free operations. Phase 3 does not widen it again. It connects what was already there. Until now every claim was verified in isolation: if one input number was wrong, the ten claims derived from it were each flagged on their own, with no record that they shared a cause. Phase 3 adds the missing structure — a dependency graph over the claims — and uses it for three things: it traces a single root error to its downstream consequences instead of reporting ten flat flags (Move 1); it weights the trust score by how much rests on each failing number, so a wrong figure fourteen claims depend on costs more than an isolated one (Move 3); and, separately, it annotates a cross-document conflict with whether the filer's own text discloses a restatement or reclassification near the disputed number (Move 2). None of the three introduces a model-judgment step into the verifier. The graph's edges, the score's weights, and the disclosure scan are all pure functions over data extraction already produced.

## Precondition that was closed first: the EPS variant guard

Phase 3's spec opens by requiring that the Phase 2 cross-statement path actually behaves as documented before anything is built on top of it. Closing that surfaced a real bug, now fixed, worth recording plainly because the earlier write-up described the opposite of the true behavior.

The `eps_reconciliation` rule checks that a stated EPS equals net income over shares. EPS comes in two variants — basic and diluted — and diluted shares are larger, so comparing a stated *basic* EPS against grounded *diluted* shares produces a number that looks wrong but isn't; it's an apples-to-oranges comparison. The §4 guard exists to refuse that comparison rather than emit a false `WRONG_MATH`.

The bug: `eps_variant_consistent()` returned "consistent, proceed" whenever the variant was *unrecorded* on either side. That let the arithmetic run on untagged operands and emit a confident `WRONG_MATH` — a verdict indistinguishable from the basic/diluted mismatch the guard was built to catch. The Phase 2 notes claimed "if variant is unrecorded, returns `AMBIGUOUS`." The code did the reverse.

The fix makes the function three-valued: `True` (both sides tagged and matching → arithmetic runs), `False` (both tagged, disagree → `AMBIGUOUS`), and `None` (either side untagged → `AMBIGUOUS`, because a `WRONG_MATH` from untagged operands can't be trusted). The verifier handles the `None` case with an explicit explanation that names the remedy: the extractor must tag `eps_variant` on the claim and `category` on the shares operand. Five existing tests were corrected — four had passed operands with no variant tags and now tag them; one, previously named `test_eps_unknown_shares_variant_does_not_block`, had asserted the *buggy* behavior and is now `test_eps_unknown_shares_variant_blocks_to_ambiguous` with the assertion flipped and a comment explaining why. One regression-guard test, `test_no_variant_recorded_is_ambiguous`, was added to pin the exact bug scenario.

The honest one-liner, now true in both code and docs: **a `WRONG_MATH` on an `eps_reconciliation` claim is correct only when the extractor tagged the variant on both sides; an untagged claim is held at `AMBIGUOUS`, never run blind.** That is a named limitation, not a solved problem — exactly the framing Phase 2 should have carried.

## Second precondition closed: the cross-document layer was built but never wired in

Testing a two-filing input (a FY2024 10-K and a FY2025 10-K that restates the prior year) surfaced a more fundamental gap than any single move, and it is worth recording in full because the symptom looked like several unrelated bugs.

The symptoms, observed across repeated live runs of the same two-document pair: a correct FY2025 net-income claim was flagged `WRONG_MATH` because it was grounded against the FY2024 document's figure; a diluted-EPS control case built to verify cleanly did the same; the cash tie-out ran only on the first document; and the cross-document revenue `CONFLICT` ($740M vs $710M "as restated," with explicit restatement language beside it) never appeared at all.

The root cause was singular: **the registry, conflict detection, and restatement classification were all built and unit-tested in `core/`, but nothing in the live pipeline ever called them.** `audit(source, summary)` took a single concatenated `source` string, so a multi-filing input became one undifferentiated blob. With no document routing, the extractor grounded same-named line items ("net income," "diluted EPS," "cash") against the first match — the older filing — and because no `DocumentRegistry` was ever constructed, `find_conflicts` and `classify_restatement` had no call site. The §7 "never silently pick a winner" guarantee was unreachable in practice, not because the detection was wrong, but because it was never invoked.

The fix is a new multi-document path, `audit_documents(documents, summary)`, that does what the single-string path structurally could not. It builds a `DocumentRegistry` from labeled documents, so a claim's operand can name which filing it came from. It runs cross-statement extraction *per document*, so each filing's cash tie-out is checked against its own cash lines. It labels each document with its id and period in the grounding prompt and instructs the extractor to route a fiscal-year claim to the document that describes it — and, critically, to ground a restated prior-year base from the later filing's restated figure rather than a stale value in an earlier one. And it calls `find_conflicts` + `classify_restatement` through a new pure-code bridge (`core/conflicts.py`) that emits a `CONFLICT` verdict per disagreement, carrying the disclosure-language annotation, folded into the score so a source disagreement lowers trust. The original `audit()` is unchanged and every prior test passes; the new path is additive.

The evidence it works is a deterministic, no-LLM test layer built on the exact reported failure case (the Vesper Materials pair). Fed claims grounded to the correct document — i.e. assuming extraction did its job — every component now returns the right verdict: the restated-base growth and the margin verify, net income and diluted EPS verify against the FY2025 figures instead of false-flagging against FY2024, the cash tie-out is caught as `WRONG_MATH` on the document with the real gap, and the FY2024 revenue conflict surfaces and is classified `EXPLICIT_RESTATEMENT`. Two new suites (25 tests) pin this end to end through `audit_documents`.

Live follow-up on the same Vesper-style Phase 3 demo surfaced a narrower extraction-shape issue, not a verifier issue: Groq sometimes returned `identity` with two operands for formulas or contextual phrases (`gross margin 35.0%` with gross-profit/revenue operands; `diluted EPS $1.36 on 82.4M shares` with the share count included as a second identity operand). The verifier correctly held those as `AMBIGUOUS`, but the output was less useful than it should be. Parser-side normalization now repairs only those two narrow shapes before verification: gross-margin identity with two operands becomes `margin_percent`, and EPS identity with contextual share-count operand keeps only the EPS operand. Two regression tests pin both cases.

Second live-result hardening: `cash_flow_tie_out` no longer emits confident `WRONG_MATH` when the extracted source context itself discloses restricted-cash / escrow / reconciling-item language explaining the gap between cash-flow ending cash and balance-sheet cash. The numeric mismatch is still surfaced with recomputed value and delta, but status becomes `AMBIGUOUS` because the issue is presentation basis, not necessarily document arithmetic. Undisclosed cash gaps remain `WRONG_MATH`. Cross-statement extraction prompt now explicitly tells the model to carry nearby restricted-cash disclosure into `source_text` or `notes`, so deterministic verifier has the text it needs.

Backend-surface demo tests now exercise the FastAPI `/audit-multi` response shape across phases with deterministic fixtures: Phase 1 verified arithmetic, Phase 2 disclosed restricted-cash ambiguity, Phase 3 propagated error (`caused_by` serialized), cross-document `CONFLICT` with `EXPLICIT_RESTATEMENT`, and weighted/unweighted score fields. A real local Uvicorn smoke test also hits `/health`, `/examples`, and bundled `/audit` replay over HTTP.

The honest boundary, restated: this fix makes the cross-document machinery *reachable and correct given correct grounding*. The remaining dependency is the same one named throughout — the extractor must actually route each claim to the right document. The new path gives it labeled documents and explicit instructions to do so, and the deterministic tests prove every layer downstream of grounding is right; live grounding accuracy on messy real filings is measured separately, not asserted here. Cross-document conflict detection also requires comparable labelled figures (table cells) for the documents; prose-only input with no parsed table structure yields no conflict rather than a guessed one — table extraction over real filings is the named next step there.

## What shipped in Phase 3

Three moves, built in the spec's dependency order (Move 1 first, because Moves 2 and 3 either build on it or run beside it). The verification path grew by three concerns across two new core modules (`core/graph.py`, `core/restatement.py`) and one rewritten one (`core/score.py`), all model-free. The test suite grew from **157 to 190 passing** — 13 new graph tests, 7 new weighted-score tests, 13 new restatement tests — and all 157 pre-existing Phase 1/2 tests pass unchanged, which is the evidence the additions are genuinely additive.

### Move 1 — Provenance graph and propagated error

The problem is that independent verification hides shared causes. A misstated revenue figure that feeds a margin claim and a growth-rate claim produces three separate flags, with nothing telling a reviewer that two of them are consequences of the first.

Move 1 adds two optional fields to `Claim` — `node_id` (a stable identifier for this claim's output value) and `depends_on` (the `node_id`s an operand was sourced from, *when that operand is itself another claim's stated output* rather than a raw figure). Both default empty, so a Phase 1 or Phase 2 claim is a leaf and behaves exactly as before. `core/graph.py` builds a directed acyclic graph from those edges, validates acyclicity at construction (a cycle is impossible from correct extraction but cheap to catch and a clean signal of an extractor bug), and exposes downstream reachability.

The propagation pass runs *after* normal verification, never instead of it. Every claim is still verified independently first. Then the graph is walked: any claim whose own verdict is `WRONG_MATH` or `UNSUPPORTED_NUMBER` is treated as a root, and every claim reachable downstream of it is relabeled `PROPAGATED_ERROR` with a `caused_by` field pointing at the root — *unless that downstream claim independently failed too*, in which case its own `WRONG_MATH` stands and is not masked. When a claim sits downstream of several distinct roots, it is attributed to the nearest one, so "wrong because of X" names the proximate cause a reviewer would chase first. The result is "one root cause, N consequences" in the output, instead of N flat flags.

The part the spec correctly flags as easy to underestimate is the extraction side: the graph does nothing unless the extractor actually tags when one claim's operand is another claim's stated output. Without those `depends_on` edges, every claim defaults to a leaf and Move 1 is inert. That tagging is extraction-prompt work, not verifier work; the graph logic trusts the edges exactly as the verifier trusts grounded operands, and the test suite builds those edges by hand to test the graph logic in isolation from extraction.

### Move 3 — Dependency-weighted score

The flat Aritiq Score weighted every claim equally regardless of where it sat in a derivation chain. With Move 1's graph in place, that can be corrected: a `WRONG_MATH` on a number many claims depend on should cost more than one on an isolated leaf.

`compute_score` now reports two numbers. The **unweighted** score is the original flat mean, kept and displayed as a secondary figure so the new weighting stays auditable and comparable. The **weighted** score weights each root claim's contribution by `base_weight × (1 + log(1 + downstream_count))` — logarithmic on purpose, so a number with fifty dependents looks *meaningfully* worse than an isolated error, not apocalyptically worse; a single failure must not crater the whole score to near zero. `PROPAGATED_ERROR` claims are excluded from scoring entirely, so a single root failure is counted once at the root rather than N times across its consequences. The intended display is the pair together — for example "Weighted 62 · Unweighted 81" — never the weighted number alone, so the score never becomes a black box.

The invariant that keeps this honest, and that is pinned by a test: with no graph structure anywhere — no `depends_on` edges — every claim's `downstream_count` is zero, the log term vanishes, and the weighted score equals the unweighted score *exactly*. The new weighting can only ever differ from the old behavior when there is real dependency structure to measure.

### Move 2 — Restatement disclosure-language classification

This is the move the spec warns most sharply about scoping, and the scope was kept narrow on purpose. Phase 2's registry surfaces a `CONFLICT` — two documents disagree on a number — but says nothing about *why*. A prior-period correction, a reclassification, and a segment realignment all look identical as a flat numeric disagreement.

The tempting version is "pull SEC EDGAR XBRL and diff the taxonomy tags to determine the restatement type." That is a large integration problem that is, in general, unsolved: tags get remapped for reasons unrelated to any real restatement (a filer switching XBRL vendors looks the same as a genuine restatement at the tag level), so an inferred restatement-type-from-tag-diff is not a claim to stand behind. Move 2 does not attempt it. The v1 that shipped is a deterministic context-string lookup: given a conflict and the text near the disputed figure, scan a bounded window for explicit restatement or reclassification language the filer itself wrote, and report which was found.

`core/restatement.py` annotates a `FigureConflict` with a `restatement_type` from a small explicit enum — `UNCLASSIFIED` (no context was available to scan), `EXPLICIT_RESTATEMENT` (the text literally says "restated" / "as restated" / "previously reported" near the figure), `POSSIBLE_RECLASSIFICATION` (reclassification or segment-realignment language is present nearby), or `UNEXPLAINED` (a real conflict with no disclosure language found near it) — and records the exact substring that triggered the match, so the annotation is auditable down to the word. Matching is word-boundary anchored against a bounded context window (200 characters by default) so that reclassification language in a faraway, unrelated sentence does not over-fire onto a plain conflict. That boundary is the single biggest risk in the move and is pinned by a test: the same keyword 600 characters from the figure stays `UNEXPLAINED`, the same keyword adjacent to it fires.

What Move 2's output means is stated narrowly and on purpose, and finished copy must follow suit: it **detects disclosure language near a conflict**. It does not determine what kind of restatement occurred. `POSSIBLE_RECLASSIFICATION` and `UNEXPLAINED` are claims about the *text* ("reclassification language is / no disclosure language is present near this number"), not determinations about the *accounting*. The XBRL angle remains a possible v2 stretch goal, and if it is ever built, the defensible output is a raw tag diff ("this figure mapped to `us-gaap:Revenues` in FY24 and `us-gaap:RevenueFromContractWithCustomer` in FY25"), reported as data — never an inferred classification asserted as solved.

## How we know it works

The discipline carried from Phase 2 holds: results are pinned by constructed-ground-truth tests with no model and no API key, and reported by what each test actually establishes rather than blended into one headline.

The graph tests build small DAGs by hand and assert exact propagation: a root failure feeding two downstream claims, one of which is also independently wrong, marks the right nodes and leaves the independently-wrong one with its own verdict; a misstated revenue figure feeding a margin claim and a growth-rate claim surfaces both as `PROPAGATED_ERROR` with the correct `caused_by`; a document with no shared operands produces an all-leaf graph and zero propagation, so there is no false grouping; and a constructed cycle raises at construction time.

The weighted-score tests pin both required properties: two documents with identical claim-level pass/fail patterns but different dependency structure produce different weighted scores in the correct direction (more dependents on a failure → lower weighted score), and a document with zero shared dependencies produces weighted equal to unweighted exactly. A separate test confirms the weighting is logarithmic rather than linear — a root with fifty dependents carries a single-digit weight, not a fifty-times one.

The restatement tests pin the three required cases — explicit "(restated)" language in the figure's context yields `EXPLICIT_RESTATEMENT`, no nearby disclosure yields `UNEXPLAINED` rather than a guess, and the over-fire boundary is tested in both directions — plus that absence of any context yields `UNCLASSIFIED` (not `UNEXPLAINED`), since "we had nothing to look at" is a different statement from "we looked and found nothing."

The firewall was re-checked after the additions: `aritiq/core/` contains zero imports of any model SDK. The three new concerns import only `math`, `re`, the standard library, and sibling core modules.

## The honest boundary, restated for Phase 3

The boundary Phase 2 named still holds and Phase 3 sharpens it. The verifier is genuinely model-free, and now so are the graph, the score weighting, and the disclosure scan — but their *inputs* are only as sound as the extractor's tags. "The graph traces error propagation correctly" is unconditionally true given correct `depends_on` edges; "the edges correctly capture which claims actually share a number" is an extraction property, and a wrong or missing edge fails silently — an unlinked shared number simply doesn't propagate, and a wrongly-linked one propagates where it shouldn't. The same applies to Move 2: the scan reports the disclosure language it finds near the figure the extractor located; if the figure was mis-located or the context was mis-extracted, the annotation reflects that, not the filing. The firewall guarantees no model grades the math, the graph, or the score. It does not guarantee the labels and edges feeding them are right. That error is measured separately and reported by type. The line is not a weakness to hide; it is the precise boundary that makes everything inside it defensible.

## What is deliberately not claimed

Three constraints carried directly from the Phase 3 spec, recorded here so finished pitch and documentation copy honor them:

No novelty claim. Neither this work nor its write-up rests on a literature search rigorous enough to assert that no prior work has done any of this, and asserting it without one would undermine the project's actual selling point, which is intellectual honesty.

No "research grade" self-description. The work is left to demonstrate its own rigor; calling it research-grade reads as exactly the kind of overclaim the project's design philosophy argues against. (Earlier internal notes used the phrase; it is not carried forward.)

Move 2 is described as detecting disclosure language near a conflict, never as "detecting restatement type." It scans for the filer's own restatement or reclassification language in a bounded window around a disputed figure. That is a narrower and far more defensible claim than determining what kind of restatement occurred, and the copy should not quietly widen it.

## Definition of done — met

The provenance graph exists, builds from `depends_on` edges, validates acyclicity, and drives a propagation pass that turns N flat flags into one root cause plus N attributed consequences — with independently-broken downstream claims never masked. The score is dependency-aware, excludes propagated consequences so a root error is counted once, weights logarithmically so a single failure is meaningful but not catastrophic, and degrades exactly to the old flat score when there is no structure to weight by — with both numbers shown together. The restatement annotation is a deterministic, bounded, word-boundary disclosure-language scan with an auditable matched-substring trail, scoped to context strings and honest about meaning "language found / not found near the figure," not "restatement type determined." The EPS precondition bug is fixed, its tests corrected, and its limitation stated truthfully. Every new concern has a one-sentence reason it is still "code verifies" and not "a model grading a model," and the suite now passes at 232 tests.
